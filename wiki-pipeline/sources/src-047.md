## Summary

Curated synthesis for process **1.3 Operational Internal Communication Norms**. Primary inputs: HN threads (src-042, src-043), Basecamp guide (src-044), Gaurav Vohra principles (src-045), Antoine Buteau channel strategy (src-046), plus `data/processes.json` stageFocus and `data/stages.json` headcount bands.

**Pipeline decision:** Do not redo atom-192–217. Use this source + atom-218 for a **thin** Phase 2 wiki (principles + stage decisions + pointers). Full artifact template lives in `docs/proposals/communication-norms-summary-and-template.md`.

**Headcount:** No agreed magic number in sources. Use stage bands + pain triggers (remote, cross-team, hire #2+ outside founding context), not a single threshold.

## Schools (cross-cutting)

| School | What channel / memory | How much written |
|--------|----------------------|------------------|
| **Chat-forward** | Slack is coordination layer; search helps; public channels | Light norms doc; codify big decisions |
| **Evergreen-first** | Email/wiki default; Slack sparingly | Wiki-first; Confluence-per-answer pattern |
| **Async writing** | Basecamp/doc comments; meetings rare | Heavy on written decisions; check-ins not chat |
| **Organic / coach** | Norms emerge in small room; post-hoc coaching | Minimal until pain; “don’t die on hills” |

**Synthesis (wiki voice):** Chat for movement, docs/tickets for memory. Write the **minimum** that prevents chaos — one page when remote or cross-team, one-liners when co-located founding circle.

## What to decide, by stage

### Foundation (0–1) — process `future`; usually skip wiki page

**Decide:** Primary chat tool? Urgent = call/DM?

**Schools:** Oral norms enough vs one main channel anyway.

**Middle ground:** One-liner in handbook: *one channel; urgent = call; non-urgent = channel; same-day if you can.* (`processes.json` stageFocus)

---

### First hires (2–10) — `recommended`; thin wiki only

**Decide:**

1. **Channel list** — which `#channels` exist and what each is for?
2. **Public vs DM default** — operational work in channels?
3. **What must leave chat** — decisions → wiki/ticket/repo?
4. **Response expectation** — instant vs ~3 working hours when tagged?
5. **Who owns** customer questions / bugs? (SLAs if CS exists)

**Schools:**

| Decision | A | B | Middle |
|----------|---|---|--------|
| Internal important news | `#announce` | Email first | Announce in chat **with link** to doc |
| Slack role | Core tool | Minimize | Coordination + link to record |
| Written norms | One-page required reading | Coach only | One page if **remote** or **≥3 non-founders** |
| Enforcement | Policy (codify Slack) | Discourage + coach | 3 non-negotiables, coach the rest |

**Triggers for more than one-liners:** remote team, first non-founder hires, CS→eng handoffs starting.

**Do not decide yet:** Full playbook, escalation matrix, on-call rotations, artifact map for quarterly OKRs (see process 1.1).

---

### Early revenue (11–25) — `critical`

**Decide:**

1. **Written channel guidelines** — what goes where, DM rules
2. **Cross-team escalation** — e.g. CS → engineering path
3. **Incident fast lane** — which channel, when `@channel`
4. **Artifact map (light)** — decisions, scope, incidents: chat notifies, X authorizes
5. **Email's job** — external + formal internal summarize-with-link

**Schools:** Same as first hires, but **written guidelines** camp gains weight; organic-only camp breaks on cross-team handoffs.

**Middle ground:** One-page norms → add escalation table + “chat for movement, not memory” with 48h codify rule.

---

### Growth (26–50) — `critical`

**Decide:**

1. **Communication playbook** — SLAs by urgency type
2. **Escalation matrix** — on-call / rotation for interrupt-driven queues
3. **Review cadence** — what’s working quarterly?
4. **Integration** — chat ↔ tickets ↔ wiki

**Schools:** GitLab-handbook depth vs keep one living doc. Sources favor **living one-pager + ticket/wiki spine**, not 2,000 pages.

---

### Scaled (51+) — `critical`

**Decide:** Full comms architecture, SLA reporting, incident protocols, tooling integration.

**Out of scope** for this repo’s thin wiki — point to external references (GitLab handbook, etc.).
