## Candidate claims from GitLab Company-Level Planning & Goals

### c-gitlab-planning-foundation-001: Strategy can start as shared context
- Section: what_good_looks_like
- Supporting atoms: atom-072-gitlab
- Candidate claim: At foundation stage, company-level planning can be a shared strategy reference plus explicit daily prioritization norms; you do not need a formal OKR process yet.
- Synthesis notes: GitLab's 2015 handbook shows the early form: strategy exists, but operational alignment is mostly in workflow rules and shared context.

### c-gitlab-planning-foundation-002: Keep planning close to work selection
- Section: what_to_do
- Supporting atoms: atom-073-gitlab
- Candidate claim: Use a simple current-milestone discipline: keep issues actionable, pick work from the current milestone, and only assign work when someone is actually starting it.
- Synthesis notes: This is a useful early-stage planning lesson because it connects planning directly to execution instead of creating a separate planning ritual.

### c-gitlab-planning-foundation-003: Protect focus with a promise rule
- Section: warning_signs
- Supporting atoms: atom-074-gitlab
- Candidate claim: If people are making external promises without internal agreement, your planning process is already leaking; add a lightweight rule that commitments need internal alignment.
- Synthesis notes: This is a warning-sign interpretation of GitLab's explicit promise rule. It should be triangulated with another operator source before publication.
